# Feature 3: Intelligent Question Generation & 4-Source Merge

---

## Overview
SmartFU generates follow-up questions through **4 independent sources**, then merges and deduplicates them through a regulatory gate that enforces an **absolute maximum of 5 questions** per follow-up. This system is the core innovation — it combines human expertise, regulatory rules, institutional knowledge, and AI intelligence in a single pipeline.

---

## 3.1 Four Question Sources

```
Source 1: REVIEWER QUESTIONS      (Priority 0 — highest)
  ├── Human PV Specialist adds questions manually
  ├── Stored in reviewer_notes with [REVIEWER_QUESTION] tags
  ├── Always included (never filtered out)
  └── UI: Blue badges in CaseAnalysis

Source 2: TFU MANDATORY            (Priority 1)
  ├── ICH E2A / EMA GVP Module VI regulatory requirements
  ├── Generated by TFU Decision Agent (tfu_rules.py)
  ├── Based on missing fields, event type, seriousness
  └── UI: Amber badges in CaseAnalysis

Source 3: REPO FORM QUESTIONS      (Priority 2)
  ├── Extracted from repository documents (TAFU forms, pregnancy forms)
  ├── Filtered by Mistral AI for case relevance
  ├── Stored in repo_documents table
  └── UI: Indigo badges in CaseAnalysis

Source 4: AI GENERATED             (Priority 3 — lowest)
  ├── LangGraph 7-agent pipeline output (Agent 6)
  ├── QuestionValueScorer with RL-enhanced scoring
  ├── Question text generated by Mistral LLM
  └── UI: Purple badges in CaseAnalysis
```

## 3.2 TFU Decision Agent (tfu_rules.py)

**File**: `backend/app/config/tfu_rules.py` (620 lines)

This is the regulatory brain. No LLM — pure rule-based logic for compliance.

### Constants
```python
ABSOLUTE_MAX_QUESTIONS = 5  # Hard cap per follow-up round

MISSING_VALUES = [
    None, "", "unknown", "not provided", "n/a", "na", 
    "none", "not specified", "not reported", "null",
    "unspecified", "*", "-", "—"
]
```

### Keyword Sets
These drive event-specific question selection:

| Set | Examples | Purpose |
|---|---|---|
| `SERIOUS_EVENT_KEYWORDS` | death, hospitalization, life-threatening, disability, congenital anomaly | Identify serious AEs per ICH E2A |
| `SPECIAL_SITUATION_KEYWORDS` | pregnancy, overdose, medication error, off-label, elderly, pediatric | Trigger special questions per EMA GVP |
| `RMP_RISK_KEYWORDS` | hepatotoxicity, QT prolongation, Stevens-Johnson, anaphylaxis, rhabdomyolysis | Known risk management plan items |

### Event-Specific Question Groups (5 groups)

| Event Type | Questions Generated |
|---|---|
| **Death cases** | Date of death, autopsy results, cause of death, concomitant illnesses |
| **Hospitalization** | Admission/discharge dates, treatment received, discharge diagnosis |
| **Life-threatening** | Emergency treatment, intensive care, vital signs at event |
| **Disability** | Nature of disability, functional impairment, reversibility |
| **Congenital anomaly** | Gestational age at exposure, trimester, pregnancy outcome |

### Baseline Regulatory Question Pool (10 questions, priority-ordered)

1. Date of onset of adverse event
2. Outcome of the adverse event (recovered, recovering, not recovered, fatal)
3. Treatment given for the adverse event
4. Whether the suspect drug was stopped/reduced (dechallenge)
5. Result of dechallenge (did the event improve?)
6. Whether the drug was restarted (rechallenge)
7. Relevant medical history
8. Concomitant medications at time of event
9. Lot/batch number of suspect drug
10. Reporter's assessment of causality

### Pipeline: `tfu_decision_agent(case_data, missing_fields)`

```
Step 1: _decide_tfu_required(case_data) → bool
    ├── Returns True if:
    │   ├── Any serious event detected
    │   ├── Missing critical fields detected
    │   ├── Risk score ≥ 0.6
    │   └── Event outcome unknown
    └── Returns False → no TFU questions

Step 2: _collect_candidates(case_data, missing_fields) → List[Dict]
    ├── Scan for serious event keywords → add event-specific questions
    ├── Scan for special situations → add situation questions
    ├── Match missing fields → add targeted questions
    └── Fill remaining from baseline pool (priority order)

Step 3: Deduplicate + score candidates

Step 4: Sort by priority score (descending)

Step 5: Slice to ABSOLUTE_MAX_QUESTIONS (5)

Return: List[{text, priority, source: "TFU_MANDATORY", regulatory_basis}]
```

## 3.3 The TFU Gate: apply_tfu_gate()

This is the **central merge function** that combines all 4 sources:

```python
def apply_tfu_gate(
    ai_questions: List[Dict],         # Source 4: AI-generated
    reviewer_questions: List[Dict],   # Source 1: Human reviewer
    tfu_questions: List[Dict],        # Source 2: TFU regulatory
    repo_questions: List[Dict],       # Source 3: Repository forms
    case_data: Dict
) -> List[Dict]:
```

### Source Priority System

| Source | Priority Value | Meaning |
|---|---|---|
| `REVIEWER_QUESTION` | 0 (highest) | Human PV specialist — always included |
| `TFU_MANDATORY` | 1 | Regulatory requirement — almost always included |
| `REPO_FORM` | 2 | Institutional knowledge — included if space |
| `AI_GENERATED` | 3 (lowest) | AI suggestion — included if space remains |

### Merge Algorithm

```
1. Start with empty merged list
2. Add ALL reviewer questions (never dropped)
3. Add TFU mandatory questions (dedup against reviewer)
4. Add repo form questions (dedup against above)
5. Add AI questions (dedup against above)
6. Semantic deduplication: remove questions with cosine similarity > 0.8
7. Sort by (source_priority ASC, score DESC)
8. Enforce ABSOLUTE_MAX_QUESTIONS = 5
9. If over 5: drop lowest-priority questions first (AI → Repo → TFU)
10. NEVER drop reviewer questions
```

### Output Format
Each question in the final list:
```json
{
    "text": "What was the date of onset of the adverse event?",
    "source": "TFU_MANDATORY",
    "priority": 1,
    "score": 0.92,
    "regulatory_basis": "ICH E2A Section 3.1",
    "field_targeted": "event_onset_date"
}
```

## 3.4 QuestionValueScorer (RL-Enhanced)

**File**: `backend/app/services/question_scoring.py` (1022 lines)

### Value Formula

$$\text{QuestionValue} = \text{CriticalityWeight} \times \text{RiskWeight} \times \text{UrgencyFactor}$$

### Components

**Criticality Weight** — Based on which field the question targets:

| Field Category | Weight |
|---|---|
| `REGULATORY_CRITICAL_FIELDS` (death date, causality, seriousness criteria) | 1.0 |
| Safety-critical fields (dechallenge, rechallenge, concomitant meds) | 0.8 |
| Important fields (medical history, onset date) | 0.6 |
| Nice-to-have fields (batch number, reporter details) | 0.3 |

**Risk Weight** — Scales up for high-risk cases:
- Risk HIGH → multiply by 1.5
- Risk MEDIUM → multiply by 1.2
- Risk LOW → multiply by 1.0

**Urgency Factor** — Time-based:
- Days to deadline ≤ 3 → multiply by 2.0
- Days to deadline ≤ 7 → multiply by 1.5
- Days to deadline > 7 → multiply by 1.0

### Reinforcement Learning Enhancement

| Constant | Value | Purpose |
|---|---|---|
| `RL_REWARD_WEIGHT` | 0.2 | Weight for historical reward signal |
| `RL_CONSTRAINT_WEIGHT` | 0.5 | Weight for regulatory constraint compliance |
| `RL_DUPLICATE_WEIGHT` | 0.3 | Penalty for duplicate/similar questions |

**RL Reward Signal**: Questions that previously led to useful answers (field filled) get positive reward. Questions that went unanswered get negative reward. This accumulates over time.

### Question Text Generation
- **NOT template-based** — uses **Mistral AI** (model: `mistral-large-latest`)
- Prompt includes: field name, case context, reporter type, medical event
- Temperature: 0.3 (consistent but not robotic)
- Output: Natural-language question tailored to the specific case

```python
async def filter_repo_questions_for_case(

```

| # of Repo Questions | Behavior |
|---|---|
| ≤ 10 | Return all (no filtering) |
| > 10 | Call `_call_mistral_filter()` to select 5-15 most relevant |

### Mistral Filter Call
- Model: `mistral-large-latest`
- Temperature: 0.3
- Max tokens: 1024
- Prompt: "Given this case with drug X and event Y, select the most relevant questions from this TAFU form"
- Output cap: 5-15 questions maximum

## 3.6 4-Source Merge in the API

**File**: `backend/app/api/routes/cases.py` — `analyze-and-send` endpoint

```python
@router.post("/{case_id}/analyze-and-send")
async def analyze_and_send(case_id: int, body: AnalyzeBody):
    """
    1. Gather reviewer questions from body.reviewer_notes
    2. Run TFU Decision Agent on case data
    3. Load repo doc questions (from body.repo_doc_ids)
    4. Run 7-agent pipeline (AI questions)
    5. Merge all 4 sources through apply_tfu_gate()
    6. Store FollowUpDecision
    7. Send via chosen channel
    """
```

### AnalyzeBody Schema
```python
class AnalyzeBody(BaseModel):
    repo_doc_ids: Optional[List[int]] = None
    reviewer_notes: Optional[str] = None
    language: Optional[str] = "en"
```

## 3.8 How to Explain to Judges

> "Questions aren't generated by a single AI. We have 4 different sources: the human reviewer who knows the case, a regulatory engine that checks ICH E2A requirements, institutional TAFU forms from the company's document repository, and the AI pipeline. All 4 are merged, deduplicated, and capped at 5 questions per round — because the more questions you ask, the less likely the reporter will respond. Our scores combine field criticality, case risk, and time urgency, and the scoring improves over time with reinforcement learning from actual response data."
